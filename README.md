## 📖 Overview

**Carat** is a conversational analytics platform that lets anyone query **250,000+ real UPI transaction records** using natural language. No SQL knowledge required — just ask your question, and Carat's AI engine translates it into executable SQL, runs it against a SQLite database, generates interactive Plotly visualizations, and returns analyst-grade summaries, all in seconds.

---

## ✨ Features

### 🗣️ Natural Language to SQL
- Type questions in plain English — the LLM generates optimized, validated SQL queries automatically.
- Supports conversational context — the last 2 exchanges are sent to the LLM for follow-up questions.
- Automatic follow-up question suggestions based on the context of each query.

### 📊 Smart Visualizations (Auto-Generated)
Every answer comes with auto-generated, publication-ready **Plotly.js** charts:

| Chart Type | When It's Used |
|---|---|
| **Vertical Bar** | Category + metric comparisons (≤ 8 items) |
| **Horizontal Bar** | Category + metric comparisons (> 8 items) |
| **Donut / Pie** | Percentages, distributions, proportions (2–12 categories) |
| **Scatter** | Correlation between two numeric columns |
| **Histogram** | Distribution of a single numeric column |

- Charts are sorted by value (highest → lowest) with value labels on every bar.
- Percentage columns auto-format with `%` suffixes on axes and labels.
- Consistent dark-themed styling with JetBrains Mono typography.

### 📈 Real-Time KPI Dashboard
- Live KPI cards: **Total Transactions**, **Success Rate**, **Average Amount**, **Failure Rate**, **Peak Hour**, **Top State**.
- Data is fetched directly from SQLite on page load via the `/stats` endpoint.
- Sidebar with pre-built "Quick Queries" for one-click exploration.

### 💡 AI-Powered Analysis Summaries
- Every query result includes a **4–6 sentence professional analyst summary** generated by the LLM.
- Summaries highlight key findings, compare top vs. bottom performers, and provide business context.
- Automatic fallback to a rule-based summary if the LLM call fails.

### 🔄 Automatic Model Fallback
- Configurable **primary** and **fallback** LLM models via environment variables.
- If the primary model fails (e.g., 402 credit error), the system auto-retries with the fallback model seamlessly.
- Both the Pandas engine and SQL engine have independent two-tier fallback logic.

## 🌟 Unique & Standout Features

### 1. 🛡️ AST-Based Safe Execution Sandbox (`planner.py` + `safe_exec.py`)
> **No `exec()`. No `eval()`. Zero arbitrary code execution.**

This is Carat's most distinctive security feature. LLM-generated Pandas code is **never executed directly**. Instead:

1. **`planner.py`** uses Python's `ast` module to parse LLM output into a structured **execution plan** (a plain dictionary).
2. **`safe_exec.py`** dispatches the plan to **pre-defined handler functions** — only whitelisted operations like `groupby`, `filter`, `mean`, `sum`, `count`, etc. are allowed.

```
User Question → LLM → Pandas Code → AST Parser → Execution Plan (dict) → Safe Handler → Result
```

Supported safe operations:
- `filter`, `filter_multi`, `filter_then_groupby`
- `groupby` (with agg functions: sum, mean, count, min, max, median, std, var, nunique, first, last)
- Single-column aggregations (mean, sum, count, max, min, median, std, var, nunique)

This architecture completely eliminates prompt injection and code injection attacks through the Pandas path.

---

### 2. 🧠 Fuzzy Entity Resolution (`engine.py`)
Carat auto-corrects **typos and aliases** in user queries before processing:

- **Alias mapping**: "Bombay" → Maharashtra, "Bangalore" → Karnataka, "Chennai" → Tamil Nadu, "gpay" → GPay, "wifi" → WiFi, etc.
- **Fuzzy matching**: Uses `difflib.SequenceMatcher` to find close matches (cutoff ≥ 0.6 for auto-correction, ≥ 0.5 for "Did you mean?" suggestions).
- **Bigram matching**: Also matches two-word phrases (e.g., "Himachal Pradesh" with a typo).

Auto-corrections are appended to the insight text so the user knows what was changed.

---

### 3. ⚡ Fast-Path Query Engine (13+ Shortcuts)
For **common high-frequency queries**, the Pandas engine skips the LLM entirely and computes results directly:

| Shortcut | Example Query |
|---|---|
| Weekend traffic % | "What percentage of transactions are on weekends?" |
| Overall fraud rate | "What is the fraud rate?" |
| High-value fraud | "Fraud rate for transactions above 5000" |
| Failure rate by device | "Failure rate by device type" |
| 5G vs WiFi failure | "Compare 5G and WiFi failure rates" |
| iOS vs Android success | "Success rate iOS vs Android" |
| 4G vs WiFi volume | "Transaction volume 4G vs WiFi" |
| State for 18–25 age group | "Which state has most transactions for 18-25?" |
| P2P by age | "P2P transfers by age group" |
| Bank failures | "Which bank has the most failed transactions?" |
| Avg amount by category | "Average amount by merchant category" |
| Peak hour (per category) | "Peak hour for Entertainment" |
| Peak hours (all categories) | "Peak hours by category" |

These fast paths deliver results **in milliseconds** with zero API cost.

---

### 4. ✅ Automated Result Verification (`verify.py`)
Every query result is validated before being returned to the user:

- **Null check**: Errors if the result is `None`, empty, or has entirely null numeric columns.
- **Filter verification**: Spot-checks that filter conditions are reflected in the first 5 rows of the result.
- **GroupBy sanity**: Warns if a groupby operation returns only 1 or fewer unique groups.
- **Row count warning**: Flags results exceeding 10,000 rows as potentially too large to display.
- **Model tracking**: Every response includes which LLM model was used and whether the fallback was triggered.

---

### 5. 📝 JSONL Failure Logging & Admin Dashboard (`failure_logger.py`)
- All AST parsing failures are logged to `logs/ast_failures.jsonl` with: timestamp, raw LLM output, original query, error message, and unsupported node type.
- An **admin-only endpoint** (`/admin/failures`) lets authorized users review the 50 most recent failures.
- This creates a self-improving feedback loop — logged failures inform prompt improvements.

---

### 6. 🗃️ LRU Cache with History-Aware Keys (`shared_utils.py`)
- Results are cached in an **LRU cache** (200 entries, `OrderedDict`-based).
- Cache keys are normalized (lowercased, stripped) and **include an MD5 hash of the last 4 chat messages** — so contextual follow-ups don't collide with identical standalone queries.
- Eviction follows true LRU semantics: the least-recently-used entry is evicted when the cache is full.

---

### 7. 🎨 Dual Response Card UI
The dashboard dynamically renders two distinct card types:

- **SQL Response Card**: Includes an `[SQL]` badge, analyst summary, Plotly chart, collapsible raw query results, and metadata (mode, row count, elapsed time).
- **Clarification Card**: Amber-themed card with a "Needs Clarification" header and alternative query suggestions when the input can't be interpreted.

Both cards are XSS-safe — all user-generated strings are sanitized via `document.createTextNode()` before injection.

---

### 8. 🔍 Intelligent Question Validation (`shared_utils.py`)
Before any LLM call, questions are validated against a **56-keyword allow-list** covering UPI domain terms (transaction, fraud, bank, state, device, etc.). Off-topic questions are rejected immediately with a helpful message, saving LLM API costs.

## 🚀 Quick Start

### Prerequisites
- **Python 3.8+**
- **OpenRouter API Key** (or OpenAI-compatible API key)

### Installation

```bash
# 1. Clone the repository
git clone <repository-url>
cd Carat-by-team-diamond-for-insightxIITB

# 2. Install dependencies
pip install -r requirements.txt

# 3. Create .env file
cat > .env << EOF
OPENROUTER_API_KEY=your_api_key_here
PRIMARY_MODEL=openai/gpt-4o-mini
FALLBACK_MODEL=openai/gpt-3.5-turbo
SQLITE_DB_PATH=carat.db
EOF

# 4. Migrate CSV data to SQLite
python migrate_to_sqlite.py

# 5. Launch the server
python server.py
```

### Access Points
| Page | URL |
|---|---|
| 🏠 Landing Page | [http://localhost:5050](http://localhost:5050) |
| 📊 Dashboard | [http://localhost:5050/dashboard](http://localhost:5050/dashboard) |

## 🔐 Security

| Layer | Protection |
|---|---|
| **SQL Injection** | Only `SELECT` / `WITH` queries allowed. 12 forbidden keywords (`INSERT`, `UPDATE`, `DELETE`, `DROP`, `ALTER`, `CREATE`, `TRUNCATE`, `GRANT`, `REVOKE`, `EXECUTE`, `COPY`, `SET ROLE/SESSION`) are regex-blocked. |
| **Code Injection** | AST-based sandbox — LLM-generated code is parsed into structured plans, never executed via `exec()` or `eval()`. |
| **XSS Prevention** | All user strings sanitized via `document.createTextNode()` before DOM insertion. |
| **Rate Limiting** | Configurable per-minute limits on query endpoints (default: 20/min) and admin endpoints (10/min). |
| **Admin Auth** | `/admin/failures` endpoint requires `X-Admin-Key` header matching `ADMIN_KEY` env var. |
| **DB Column Validation** | `db.py` validates column names against `PRAGMA table_info` before constructing queries. |
| **Input Validation** | Questions must be ≥ 10 chars and contain at least one domain-relevant keyword. |

## 🔧 Troubleshooting

| Issue | Solution |
|---|---|
| **Port conflict** | App uses port 5050 to avoid conflicts with common Windows services |
| **402 API Error** | Check your OpenRouter credits — the system will auto-retry with the fallback model |
| **404 API Error** | Verify model names in `.env` are valid OpenRouter model identifiers |
| **429 Rate Limit** | Wait or increase `RATE_LIMIT_PER_MINUTE` in `.env` |
| **Empty database** | Run `python migrate_to_sqlite.py` to populate from CSV |
| **Missing `flask-limiter`** | Rate limiting degrades gracefully — install with `pip install flask-limiter` |
